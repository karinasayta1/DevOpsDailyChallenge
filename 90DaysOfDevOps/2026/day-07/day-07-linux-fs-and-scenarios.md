# 🚀 Day 07 – Linux File System & Scenario-Based Practice

---

## 📁 Core Directories

### / (Root)
The base of the entire file system. All other directories branch from here.  
I would use this to navigate the Linux file system from the top level.

---

### /home
Home directories of users. Contains personal directories for each user. Users can access and manage their own files here without affecting others.  
I would use this to access or manage user-specific data.

---

### /root
Home directory of the superuser (root). Separate from normal users for security purposes.  
I would use this to store files that should only be accessed by the root user.

---

### /etc
Contains configuration files of services, applications, and the system itself. Also includes configuration files related to users and system settings.

- /etc/environment - Used to define global environment variables accessible to all users. I would use this to set system-wide environment variables.  
- /etc/adduser.conf - Contains default configuration for creating new users such as shell, home directory, and group settings. I would use this to customize default user creation behavior.

---

### /var/log
Contains log files generated by the system, services, and applications. These logs are critical for troubleshooting, monitoring, and auditing.

---

### /tmp
Temporary files created by applications, services, or users are stored here. Files are usually deleted after reboot.  
I would use this to store temporary data that is not needed permanently.

---

## 📁 Important Supporting Directories

### /bin
Essential commands (ls, cp, mv). Often linked to /usr/bin.

---

### /usr
“User system resources”

Subfolders:
- /usr/bin → user commands  
- /usr/lib → libraries  
- /usr/share → shared data  

---

### /opt
Reserved for installing optional 3rd party software that are not part of core system distribution (ex. google chrome).  
I would use this directory to see what external or commercial applications are installed.

## ⚙️ Scenario-Based Practice (Advanced Thinking)

---

### 🔥 Scenario 1: Service Not Starting

systemctl status myapp - To check whether the service is running, stopped, or failed and see immediate error messages.  

journalctl -u myapp -n 50 - To view the last 50 logs of the service and identify the root cause of failure.  

systemctl is-enabled myapp - To check if the service is configured to start automatically on system boot.  

systemctl restart myapp - To restart the service after fixing the issue or to verify if it can start successfully.  

netstat -tulnp | grep myapp - To check if the required port is already being used by another process.  

---

### 🔥 Scenario 2: High CPU Usage

top - To monitor real-time CPU usage and identify high resource-consuming processes.  

ps aux --sort=-%cpu | head - To list processes sorted by CPU usage and find the top consumers.  

top -p PID - To monitor a specific process continuously using its PID.  

kill -9 PID - To forcefully terminate a process that is consuming excessive CPU or is unresponsive.  

strace -p PID - To trace system calls and understand what the process is doing internally.  

---

### 🔥 Scenario 3: Logs Deep Dive

journalctl -u docker -n 100 - To view the last 100 logs of the docker service for recent issues.  

journalctl --since "1 hour ago" - To filter logs and view only recent entries within a specific time range.  

tail -f /var/log/syslog - To monitor system logs in real-time as new logs are generated.  

grep "error" /var/log/syslog - To quickly filter and find error-related entries in log files.  

---

### 🔥 Scenario 4: Permission Issue

ls -l - To check file permissions and ownership details.  

chmod +x file.sh - To add execute permission so the script can be run.  

chown user:user file.sh - To change ownership of the file to the correct user and group.  

sudo -l - To check what commands the current user is allowed to run with sudo.  

---

### 🔥 Scenario 5: Disk Space Full

df -h - To check overall disk space usage and identify which partition is full.  

du -sh /* 2>/dev/null | sort -h - To find which top-level directories are consuming the most space.  

du -sh /var/* 2>/dev/null | sort -h - To drill down further and check which folder inside /var is large.  

du -sh /var/log/* 2>/dev/null | sort -h - To identify large log files that may be filling disk space.  

rm -rf /path/to/unnecessary/file - To delete unwanted files and free up disk space.  

journalctl --vacuum-time=7d - To clean old logs and retain only recent logs (last 7 days).  

---

## 🎯 DevOps Mindset

When something breaks, ALWAYS follow:

1. Check status  
2. Check logs  
3. Check config (/etc)  
4. Check resources (CPU, memory, disk)  
5. Restart / fix
